Agent 工程实战手册

写给想入门智能体工程的传统程序员。

ACT I · 认知篇 — 打破三个旧认知

01Agent = Model + Harness

这是整本手册的锚点。它改变了"程序员该关注什么"的答案。

很多人讨论 Agent 时，注意力都在"用哪个模型"。但真正决定 Agent 表现的，是模型之外的一切——工具定义、编排逻辑、上下文管理、错误处理、验证循环。这些统称为 harness（脚手架）。

"A decent model with a great harness beats a great model with a bad harness."
—— Addy Osmani

实证数据支持这个判断：在 TerminalBench 基准测试中，仅更换 harness 设计（模型完全不变），Agent 排名可以变化 20 多个位次。换句话说，你花在优化 harness 上的每一小时，比花在挑选模型上的回报高得多。

原则模型之外皆是工程
为什么
模型是黑盒，但 harness 是白盒——工具、错误处理、上下文、循环，每一项都是工程师可以掌控、可以优化、可以回归测试的。
怎么做
遇到 Agent 表现不好，先别想着换模型，先查 harness：工具描述够不够清楚？错误信息可不可操作？上下文给得够不够？循环次数够不够？这四个问题查完，80% 的问题能定位。
反模式
迷信"换个更强的模型就好了"。模型升级解决的是能力上限，harness 解决的是能力发挥——两者是不同维度的问题。

脚手架的共生演化

一个更深的认知：harness 不是越多越好。它有两层含义：

  • 该拆的要拆：harness 编码了对模型能力的假设。模型升级后，旧假设失效，对应的脚手架（比如过度详细的步骤拆解、冗余的格式约束）应该移除，否则越来越臃肿。
  • 该测的要测：模型是配合特定 harness 做后训练的。改动工具实现可能因紧耦合导致性能退化——改了 harness 必须回归测试。

把 harness 当成一个需要维护、需要回归的工程资产，而不是"加上去就不用管"的一次性配置。

02认知四阶：Prompt → Context → Harness → Loop

传统程序员切入 Agent 工程，认知需要经历四个阶段。这是关注点"从模型逐渐外移到系统"的过程。

大多数人学 Agent，第一反应是研究"怎么写 prompt"。这只对了第一步。完整的路径是四阶递进，越往后越是 Agent 工程的核心，也越是大多数人忽略的部分。

图 1 · 认知四阶：关注点从模型逐渐外移到系统。越往右越是 Agent 工程的核心。
阶段关注点典型问题成熟标志
阶 1
Prompt		 怎么跟模型说话 "怎么写 prompt 让模型听话" 会写 few-shot、能控制输出格式
阶 2
Context		 给模型什么信息 "模型需要哪些上下文才能决策" 会分层注入（系统/记忆/动态上下文）
阶 3
Harness		 模型怎么与系统交互 "工具怎么设计、错误怎么返回" 工具描述当 prompt 写、错误信息可操作
阶 4
Loop		 执行循环怎么设计 "状态机还是自决循环、触发和响应怎么分离" 知道"保持循环简单，复杂度下沉到工具"
自检 · 你在哪一阶？
  • 还在纠结"怎么写 prompt 让模型输出想要的格式" → 阶 1
  • 开始关心"模型需要哪些上下文才能决策" → 阶 2
  • 开始设计工具、关心错误返回的格式 → 阶 3
  • 开始质疑"我的循环是不是太复杂了" → 阶 4
关键洞察：大多数人停在阶 1，少数人进阶到阶 2，更少的人意识到阶 3 和阶 4 才是 Agent 工程的核心。传统程序员的优势在于——阶 3、阶 4 涉及的系统设计，正是你们熟悉的地盘。

03颗粒度 × 验证：双轴模型

智能体项目成败的核心变量，不是技术栈、不是模型选择，而是这两个轴的组合。

智能体时代的颗粒度失控，比传统开发更危险——因为智能体的产出是概率性的。你让程序员写一个函数，错了你一眼能看出来；你让 Agent 一次性产出整个系统的代码，错了你连看哪一行都找不到。

图 2 · 颗粒度 × 验证四象限。智能体时代的理想区与传统开发相反。
核心洞察智能体时代的理想区是"低颗粒度 × 强验证"

这跟传统开发完全相反。传统开发的理想态是"高颗粒度 × 强验证"——大需求 + 完整测试。但智能体产出是概率性的，高颗粒度下验证成本会指数爆炸（你要验证的组合太多了）。

所以策略应该是：把需求拆小，每个小块用 eval / 部署 / 回滚来验证。

自检 · 你的项目在哪个象限？
  • 你交付的单元，是"一个完整功能"还是"一个被测试覆盖的行为"？
  • 出了问题，你能定位到哪一步吗？能回滚吗？
  • 你的测试，是"跑通一次"还是"跑 5 次看通过率"？
  • 如果三个都答"前者"，你可能在危险区或安全区边缘
最可迁移的一句话：在智能体时代，颗粒度控制不是"把大需求拆成小需求"这么简单，而是"让每个交付单元都自带验证能力"——自带 eval、自带回滚、自带验收标准。
ACT II · 实践篇 — 怎么动手造 Agent

04工具设计原则

工具（function calling）是 Agent 与世界交互的接口。工具设计得好，Agent 能力上限就高。这里有六条原则中最关键的三条。

传统程序员把 API 文档当"给人看的说明"，但 Agent 时代，工具描述是给模型看的 prompt。描述得好，模型调用准确率能涨十几个百分点。

原则 1工具描述就是 prompt engineering
为什么
实测数据：在工具定义中加入具体 usage examples，准确率从 72% → 90%。工具描述不是文档，是 prompt 的一部分。
怎么做
每个工具描述都包含：做什么、什么时候用、什么时候不用、至少一个 usage example。像写 few-shot prompt 一样写工具描述。
反模式
只写一句"查询用户信息"，没有任何示例。模型只能靠猜，调用时机经常错。
原则 2返回有意义的内容，不是 ID
为什么
LLM 要基于返回值做下一步决策。返回 UUID 它看不懂，返回自然语言摘要它能直接用。
怎么做
差的设计：{"id": "abc123"}
好的设计：{"id": "abc123", "summary": "已创建项目'周末出游'，包含3个待办事项"}
反模式
返回纯 ID 或纯状态码，逼 Agent 再调一次工具去查"这个 ID 是什么"。
原则 3错误信息必须是可操作的
为什么
LLM 拿到好的错误信息可以自我修正，差的信息只能重试或放弃。这是 Agent 能否自纠错的关键。
怎么做
错误信息必须包含"原因 + 当前输入 + 建议修正"。
差："Error: invalid arguments"
好："日期格式错误：你输入了'明天'，请用 YYYY-MM-DD 格式，例如 2026-06-16"
反模式
错误信息是给人看的（"出错了，请联系管理员"），对模型毫无帮助。
传统程序的错误信息是给人看的，Agent 时代的错误信息是给模型看的——必须让模型能基于错误信息修正下一步动作。这个认知转变至关重要。

05循环设计：保持简单

所有成熟 Agent 项目的核心循环都出奇地简单。复杂度应该下沉到工具和提示词，不在循环层面做状态机。

研究主流 Agent 框架（ReAct、Plan-and-Execute、Reflexion、事件驱动、图执行、对话循环）后，最重要的结论是：

"所有成熟项目的核心循环都出奇地简单……不要在循环层面做复杂的状态机。复杂度应该下沉到工具和提示词。"

一个清晰的判断标准：如果你的 Agent 需要复杂的状态流转图，多半是工具设计不够好。改善工具描述、让错误信息更可操作，往往能让复杂状态机简化成单一 ReAct 循环。

触发 ≠ 响应

传统聊天机器人的逻辑是"触发即全流程响应"——被 @ 了就走完整流程。但成熟的 Agent 应该把这两件事分开：

原则三层决策模型
  1. 需要关注吗？（触发判断：这条消息跟我有关吗）
  2. 需要回复吗？（响应判断：要不要说点什么）
  3. 需要做复杂处理吗？（处理深度：要不要调工具、查记忆、走完整流程）

这三层解耦后，Agent 的行为更接近真实助手——可以"关注但不回复"（群里有人 @ 它但它判断不需要插话），也可以"回复但不做复杂处理"（简单问答回一句就行）。不是每次被叫到都要大动干戈。

06eval-driven 开发：Agent 时代的 TDD

传统 TDD 是"先写测试再写代码"，Agent 时代的对应物是"先写 eval 用例再开发功能"。

传统单元测试测的是"函数返回值对不对"。但 Agent 的产出是行为链路（输入 → 意图判断 → 工具调用 → 记忆存储 → 最终回复），必须用行为级测试来定义"预期"。

方法eval-driven 的三个要点
先写 eval
开发任何 Agent 功能前，先写一个 YAML/JSON 描述"输入什么、预期 Agent 怎么回应"。哪怕只写一个正向用例，也比"写完功能再想怎么测"强一百倍。
非确定性处理
LLM 是概率性的，单次测试没意义。每个用例跑 5 次，80% 通过（4/5）算过。确定性高的维度可降为 3 次，LLM 深度参与的保持 5 次。
三种判定
① 代码断言（快速客观：响应是否存在、记忆条数对不对）② LLM 裁判（灵活：让独立 LLM 按 rubric 打分）③ 人工抽查（金标准）。三种对应不同可信度需求。

六维评价体系

一个完整的 Agent eval 应该覆盖行为链路的每个环节：

  • 触发边界：该响应时响应、该沉默时沉默
  • 意图分类：empty / operational / conversational 是否判对
  • 记忆质量：存储准确性、合并正确性
  • 回复质量：任务完成度、角色一致性、简洁性
  • 工具使用：只看结果对不对，不限制调用路径
  • 副作用安全：数据库操作是否安全、有无越权

07双 Agent 协作：当能力互补时

多 Agent 不是银弹。只有当任务确实需要不同专业能力，且这些能力无法在单 Agent 中组合时，才值得引入。

"多 Agent 不是银弹。"普林斯顿 NLP 研究表明：在 64% 的基准测试任务中，单 Agent 匹配或优于多 Agent 系统。
—— 只有当任务需要不同的专业能力（如 PM + 编码），且这些能力无法在单 Agent 中有效组合时，才值得引入多 Agent。

什么时候该引入多 Agent？

判断标准是能力是否 genuinely 互补。比如：

  • 一个 Agent 擅长中文语境理解和对话协调（适合 PM 角色）
  • 另一个 Agent 擅长编码和文件操作（适合开发角色）

这种情况值得拆。如果只是"一个 Agent 慢，拆成两个加速"——不值得，反而引入了协调成本。

图 3 · 人-Agent-CC 三层协作模型。三层职责和介入频率不同。

五要素：缺一不可

如果决定引入多 Agent，这五个要素必须都做到：

  1. 能力互补（不是为多而多）
  2. 上下文解耦（给指针不给内容——传文件路径，不传整个文件）
  3. 异步 + 回调（子 Agent 执行慢，不阻塞主流程）
  4. 双层产出验证（自动化检查 + LLM 裁判打分）
  5. 边界感知权限（工作区内自由，越界必审批）
三准则子 Agent 是工具，不是同事
准则 1
传入结构化任务规范，返回结构化结果。不要期望子 Agent 理解主 Agent 的完整上下文——解耦上下文 = 子 Agent 可替换、可独立测试。
准则 2
交接用 Schema，不用自然语言。brief.md / plan.md 都是结构化文档，不是"你帮我做一下那个东西"。Schema 防止格式漂移。
准则 3
产出必须可验证、失败必须可重试。子 Agent 的产出不能直接信任，必须有独立的验证机制（自动化检查 + LLM-as-Judge 双层）。

08驾驭曲线：四阶段

你和智能体的协作关系，会经历"许愿 → 划边界 → 驾驭 → 元驾驭"四个阶段。判断自己处于哪一级，看四个物证标志。

图 4 · 驾驭曲线：从"向 Agent 许愿"到"让 Agent 驾驭 Agent"
级别名称指令文件性质权限特征规划方式
L1许愿式系统说明书 / 无 全开一次性大需求
L2边界式开始有边界 粗放有"非目标"清单
L3驾驭式有行为约束 / 开发哲学 白名单分阶段 + 验收点
L4元驾驭式约束已内化为 Agent 系统提示 由 Agent 系统管理Agent 自己拆解 + 验收
自检 · 你在 L 几？
  • 你的 prompt / 指令文件里，有没有"禁止做什么"？
  • 你给 Agent 的权限，是不是精确到具体命令？
  • 你的需求，是"一次大块"还是"分阶段、每阶段可验证"？
  • 你有没有让一个 Agent 去监督另一个 Agent？

四个都答"没有" → L1；四个都答"有" → L4。

ACT III · 避坑篇 — 别走这些弯路

09反模式：憋大招

最常见的失败模式。把 Agent 当代码生成器，一次性甩大需求，零验收。

症状一次性产出完整系统

典型表现：

  • 需求一次性甩出，没有分阶段
  • 规划文档宏大（一口气排好几周的 Phase 1-7）
  • 权限全开，包括危险操作
  • 根目录堆满 debug 文档和临时验证脚本

后果：大面积基础缺陷（比如十几个数据操作都漏了提交动作，导致数据写不进去）。错误无法定位——任何一处出错都会连锁崩溃。项目跑不起来，但 debug 文档越写越多。

让每个交付单元都自带验证能力。开新项目时先问自己："我这次的交付单元是什么？每个单元怎么验证？" 答不出"怎么验证"说明颗粒度还太大。自带 eval、自带回滚、自带验收标准——这三样是交付单元的标配。

10反模式：确定性思维陷阱

传统程序员习惯了"确定性"——代码跑对了就一直对。但 LLM 是概率性的，同样的输入可能不同输出。

症状出错才处理，而非前置防御

典型表现：

  • LLM API 调用没有超时配置——不响应时阻塞好几分钟，整个服务卡死
  • 503 / 超时直接丢消息，没有重试
  • 测试跑通一次就算过，不知道下次可能不一样
  • 信任 LLM 输出，不做独立验证

后果：服务卡死、消息丢失、偶发 bug 无法复现。企业级应用的第一要求"不丢数据、不卡死"全踩了。

把"确定性验证"重建为"概率性兜底"。每接一个外部依赖（LLM API、数据库、第三方服务），第一件事就是配超时 + 重试，不是写完功能再补。测试用统计方法判定（跑 5 次、80% 通过算过），不用单次判定。LLM 的输出必须有独立验证（LLM-as-Judge 或代码断言），不能直接信任。

11反模式：空壳闭环

功能代码写了，但没接通——有接口从没被调用，有告警服务从没接入生产链路。

症状有代码无闭环

典型表现：

  • 告警推送服务写好了，但从没接入定时任务
  • 决策记录接口定义了，但业务代码从没调用它
  • 晨报能生成，但从没流入对话

后果：看似功能完整，实则残缺。审计可追溯性（企业合规刚需）停留在接口层。查库一看，决策表是空的——整个模块是空壳。

这是 AI 协作的一个隐性风险：单点能力强，全局串联弱。在 Agent 协作下，单个功能（写个推送服务）很容易交付，但把这些功能串成闭环反而容易被遗漏——因为"串"需要全局视角，而 Agent 倾向于"你让我做啥我做啥"。必须由人主动做"串联审查"：定期检查"上个功能接没接通"，而不是假设"写了就通了"。

12反模式：无质量门禁

测试靠人肉、lint 靠自觉、明知有缺陷也无法阻止它进入主分支。

症状质量全靠人

典型表现：

  • 没有 CI——提交时不会自动跑测试和 lint
  • 测试只覆盖算法层，没有集成测试和端到端测试
  • 明知有问题（写进了待办），但因为没门禁，缺陷照样进主分支
  • 评测结果靠 log 肉眼比对，没形成回归基线

后果："能跑的 demo"和"企业级可维护项目"最明显的分界。企业级项目最基本的要求是"每次提交都经过验证"。

配 CI（哪怕只是 GitHub Actions / pre-commit）：提交时自动跑测试 + lint + 类型检查，失败阻止合并。这件事成本低、收益大，是性价比最高的工程投入之一。评测框架要工程化——用例自动跑、结果自动比对、生成回归基线，而不是靠 log 肉眼判断。

13反模式：迷信复杂状态机

想做 Agent，第一反应是"设计一个复杂的状态机/工作流"。但成熟项目的共识是相反的。

症状循环层面堆复杂度

典型表现：

  • 给 Agent 设计了多阶段固定流程（Phase 1 意图 → Phase 2 记忆 → Phase 3 回复）
  • 简单请求被强制走完三阶段，复杂请求又受限于三阶段的结构
  • 两条路径共享记忆但不共享上下文，上下文是断裂的

后果：要么完全不响应，要么全流程响应，没有中间状态。Agent 行为僵化，不够"像助手"。

保持循环简单（ReAct），复杂度下沉到工具和提示词。用 LLM 自决循环代替固定状态机——简单请求一次就回，复杂请求多轮工具调用。结构只保证边界（触发的必回、不触发的不回），复杂度交还给模型。改善工具描述和错误信息，往往能让复杂状态机简化成单一循环。

带回家清单

如果只能记住六条经验，就记这六条。每条都配一句提炼。

01
Agent = Model + Harness

模型之外皆是工程。遇到问题先查 harness 再换模型——工具、错误、上下文、循环，这四项查完能定位 80% 的问题。

02
颗粒度 × 验证双轴决定成败

智能体时代的理想区是"低颗粒度 × 强验证"。让每个交付单元都自带验证能力——自带 eval、自带回滚、自带验收标准。

03
eval-driven 是 Agent 时代的 TDD

先写 eval 用例再开发功能。跑 5 次、80% 通过算过。代码断言 + LLM 裁判 + 人工抽查，三种手段对应不同可信度。

04
子 Agent 是工具，不是同事

多 Agent 不是银弹（64% 任务单 Agent 更优）。能力互补才引入。传结构化任务、收结构化结果、独立验证产出。

05
确定性思维需重建为概率性兜底

结构类能力（分层/日志/迁移）全可迁移，但"出错才处理"的确定性思维必须重建为"前置防御"（超时+重试+熔断）。

06
用 Agent 驾驭 Agent，是时代的杠杆

当协作成熟度到 L3 之后，下一步不是"更熟练地驾驭"，而是"把驾驭这件事外包给一个专门的 Agent"。人从执行者变成架构师。

智能体时代，决定一个工程师价值的，不再是"能写多少代码"，
而是"能驾驭多少智能体"。

这条路没有终点，但有方向：
从用 Agent，到驾驭 Agent，到让 Agent 驾驭 Agent。
